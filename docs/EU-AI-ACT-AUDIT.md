## 1. Risk Classification

LETHE is an **AI-powered operating system agent** that:
- Runs locally on a user's device
- Routes to cloud LLMs when configured by the user
- Makes no autonomous decisions affecting health, safety, employment, or law enforcement
- Does not perform biometric identification, social scoring, or behavioral profiling

**Classification: Limited risk (not high-risk) — IF modules stay within scope**

LETHE's core agent does not fall under Annex III high-risk categories. It is not used
for biometric identification, critical infrastructure control, education scoring,
employment decisions, law enforcement, migration assessment, or democratic process
manipulation.

The primary obligations are **Article 50 (Transparency)** and **Article 4 (AI Literacy)**.

### Classification risk from protection modules

**This classification holds ONLY if Bios and PreuJust stay within their defined scope.**

| Module | Current scope | Safe? | What would trigger high-risk (Annex III) |
|--------|--------------|-------|------------------------------------------|
| **Bios** (Health) | Wellness info, medical card, sensor permissions, data access | Yes | Diagnosing conditions, interpreting lab results, recommending medications, predicting disease → Category 10 (health AI) → **also triggers EU MDR (medical device)** |
| **PreuJust** (Money) | Phishing detection, permission audit, scam screening | Yes | Personalized investment advice, creditworthiness assessment, bank account access → Category 6 (financial services AI) → **also triggers MiFID II** |

**The open-source exemption (Art. 2(12)) does NOT apply to high-risk systems.** If either
module crosses into high-risk territory, LETHE would need: conformity assessment, CE
marking, risk management system, technical documentation, EU database registration, and
potentially a Notified Body review.

**Mitigation:** Every new Bios/PreuJust feature must be evaluated against this table
before implementation. If a feature could cross the line, it requires legal review first.
See `docs/research/legal-compliance.md` for the full jurisdiction-by-jurisdiction analysis.

## 2. Article 50 — Transparency Obligations

### 2.1 Disclosure of AI interaction

> Providers shall ensure that AI systems intended to interact directly with natural persons
> are designed and developed in such a way that the natural persons concerned are informed
> that they are interacting with an AI system.

| Requirement | Status | Detail |
|---|---|---|
| User knows they're talking to AI | PARTIAL | UI labels messages `from-lethe` with distinct styling, agent identifies as "I am LETHE, the system" — but no explicit "You are interacting with an AI" disclosure |
| First-boot disclosure | MISSING | Wizard introduces LETHE but never explicitly states "LETHE uses artificial intelligence to generate responses" |
| Per-message AI attribution | MISSING | Messages lack a visible "AI-generated" or model attribution label |

**Required fixes:**
- [ ] Add explicit AI disclosure to first-boot welcome screen: "LETHE uses artificial intelligence to respond. Responses are generated by language models, not a human."
- [ ] Add a persistent "AI-generated" indicator on LETHE messages in the chat UI (subtle but present — e.g., small label or icon on message bubbles)
- [ ] Show which provider/model generated each response (local model name or cloud provider)

### 2.2 Synthetic content labeling

> Providers of AI systems that generate synthetic audio, image, video or text content shall
> ensure that the outputs are marked in a machine-readable format and detectable as
> artificially generated or manipulated.

| Requirement | Status | Detail |
|---|---|---|
| Machine-readable marking of AI output | MISSING | No metadata tagging on generated text |
| Detectability of synthetic content | MISSING | No watermarking or provenance marking |

**Required fixes:**
- [ ] Add metadata to LETHE responses (e.g., `data-generated-by="ai"` attribute on message elements)
- [ ] If LETHE ever generates images/audio (TTS), those outputs need synthetic content markers

### 2.3 Deepfake disclosure

Not applicable — LETHE does not generate synthetic media impersonating real persons.

## 3. Article 4 — AI Literacy

> Providers and deployers of AI systems shall take measures to ensure a sufficient level of
> AI literacy of their staff and other persons dealing with the operation and use of AI systems.

Since LETHE is deployed directly to end users (not enterprise staff), this translates to:

| Requirement | Status | Detail |
|---|---|---|
| User understands AI limitations | MISSING | No disclosure that AI can hallucinate, has knowledge cutoffs, or make errors |
| User understands data flow | PARTIAL | Provider settings show local vs. cloud, but no explicit "your messages are sent to [provider]" warning |
| User can make informed choices | PARTIAL | Provider selection exists, but tradeoffs (privacy vs. capability) are implied, not stated |

**Required fixes:**
- [ ] Add limitations disclaimer accessible from chat UI: "LETHE's responses are AI-generated and may be inaccurate, incomplete, or outdated"
- [ ] Add explicit data flow warning when configuring cloud providers: "Messages will be sent to [Provider] servers for processing"
- [ ] Document AI capabilities and limitations (see Section 6)

## 4. Article 13-15 (via GDPR interaction) — Data Processing

The AI Act does not replace GDPR. When LETHE sends user messages to cloud providers, GDPR applies.

| Requirement | Status | Detail |
|---|---|---|
| Privacy policy | DONE | `PRIVACY.md` — includes geographic restrictions, module disclaimers, data flow |
| Data processing disclosure | MISSING | No statement about what data is collected, by whom, how long |
| User rights (access, deletion, portability) | PARTIAL | Burner mode gives de facto deletion, but no formal rights mechanism |
| Consent for cloud data transmission | MISSING | No explicit consent when enabling Anthropic/OpenRouter |
| Data processing agreements with providers | MISSING | No DPA references or links |

**Required fixes:**
- [x] Create `PRIVACY.md` covering: data collected, data flow (local vs. cloud), retention (ephemeral by default), user rights, provider data handling, geographic restrictions, module disclaimers
- [ ] Add consent screen when user first configures a cloud provider: "Your conversations will be sent to [Provider]. [Link to provider's privacy policy]. Continue?"
- [ ] Link to cloud providers' data processing terms from settings

| Strength | AI Act Relevance | Source |

## 10. Summary

**LETHE's philosophy is deeply aligned with the AI Act's intent.** The boundaries system, ephemeral design, no-manipulation rules, and user sovereignty are exactly what regulators want to see. The gaps are **formal and presentational**, not architectural:

1. **Say it explicitly** — LETHE acts transparently but doesn't always *tell* users it's AI
2. **Document it** — Privacy policy, terms, and AI transparency notice are missing
3. **Get consent** — Cloud provider data flow needs explicit user acknowledgment

The architecture doesn't need to change. The UI needs labels. The project needs legal documents.
