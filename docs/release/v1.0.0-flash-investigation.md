## NEW (2026-05-04, post-boot validation): Tor + iptables not running

**On-device boot validation revealed a release-blocking gap.** The OTA installs and the property layer is intact — every `persist.lethe.*` is set, init services register, configs land in `/system/extras/lethe/` — but the network-hardening features that the LinkedIn post centers on are **not actually running**:

```
init.svc.lethe_tor       = restarting (crash loop)
avc: denied { execute_no_trans } for path="/system/bin/tor"
                          scontext=u:r:init:s0
                          tcontext=u:object_r:system_file:s0
init.svc.lethe_tor_rules = stopped (oneshot — could not write iptables either)
```

Root cause: the build's compiled `file_contexts.bin` and loaded sepolicy contain **no LETHE labels at all**:

```
$ unzip -p Lethe-1.0.0-t0lte.zip file_contexts.bin | strings | grep -iE 'lethe|tor'
(empty)
$ adb shell 'cat /sys/fs/selinux/policy | strings | grep ^lethe'
(empty)
```

The `tor` binary on disk is labeled the default `u:object_r:system_file:s0` instead of `u:object_r:lethe_exec:s0`, so init's `execute_no_trans` is denied (correctly — without the policy rule from `sepolicy/lethe.te`, init has no permission to execute it).

The repo's source policy is **correct**: [sepolicy/lethe.te](../../sepolicy/lethe.te) defines the `lethe` domain and `lethe_exec` type, [sepolicy/file_contexts](../../sepolicy/file_contexts) labels `/system/bin/tor` and the iptables script as `lethe_exec`, and `apply-overlays.sh` step 12 calls `install-sepolicy.sh` to copy them into the LineageOS source tree. So the failure is at the *build-system* level: install-sepolicy.sh's target dirs (`device/lineage/sepolicy/{common,vendor}`) likely don't exist on cm-14.1 (they're cm-12-style paths), and the script's `if [ -d "$target" ]; then` falls through silently to a fallback that the cm-14.1 `BOARD_SEPOLICY_DIRS` macro doesn't pick up. **Same family as bug #1 from the build notes** (silent-ship-nothing) — copy succeeds, build skips.

This is the same SELinux-execute issue that build-notes #2 (`capabilities` keyword) was trying to dodge by setting `seclabel u:r:init:s0`. The seclabel workaround keeps the service in the init context, but init *still* can't `execute_no_trans` a `system_file`-labeled binary into itself — it needs the `lethe_exec` label loaded plus an `allow init lethe_exec:file execute_no_trans` rule, both of which are present in the source policy and absent in the shipped policy.

**Impact on the LinkedIn post**: the post claims *"All traffic forced through Tor by iptables, UDP dropped."* As shipped, **neither Tor nor the iptables rules execute**. Networking on the device works in the clear. This claim must either be backed by a real fix or pulled from the post before publication.

What does still work on the as-shipped build (verified on-device):

- `ro.build.display.id=LETHE-1.0.0`, `ro.lethe=true/version=1.0.0`
- All `persist.lethe.*` props (sensor permissions, MAC randomization config, clipboard isolation, mic/cam indicators, PIN scramble, burner-mode flag)
- LETHE init services registered (burner, settings, tor, tor_rules — burner and settings reach `stopped` cleanly)
- LETHE bootanimation.zip on `/system/media/`, configs in `/system/extras/lethe/`
- Tracker hosts file, debloat, no Google services — all LineageOS-side and intact

What does *not* work as-shipped:

- Tor transparent proxy (SELinux blocks execve)
- iptables transparent-proxy / UDP-drop rules (same SELinux family of denials, plus init's iptables write requires the rules to actually fire from a labeled context)
- Anything else gated on a `lethe_exec`-labeled binary executing — agent and ipfs are out of scope for v1.0 anyway, but any v1.1 service inherits this same build-system gap

## NEWER (2026-05-04, after the May-4 rebuild): burner-wipe parent-dir bug

**The May-4 build still crash-loops zygote — different cause.** With the d82e555 props removed, the zygote-killer combo is gone. With sepolicy compiled in, Tor can finally execute. But once `lethe_burner_wipe` actually runs at `post-fs-data`, it does this:

```sh
rm -rf /data/app /data/data /data/user /data/user_de
```

Removing those four directories *as directories* leaves `installd` unable to recreate per-package data dirs at the next phase. `system_server` then crashes during `PackageManagerService` init, init kills `zygote` (because its child died), zygote restarts, system_server crashes again — for ~85 minutes (5163 s) until the device's battery couldn't sustain the loop.

Verified on the device: after the failed boot, mounted /data from TWRP and `ls /data/data` / `/data/user/0` / `/data/user_de/0` / `/data/app` all returned 0 entries — and the parent dirs themselves were empty (or had been recreated by some later init pass without their expected contents).

### Why May-2 didn't crash on this

The May-2 (no-sepolicy) artifact had `persist.lethe.burner.enabled=true` in its build.prop and shipped the same wipe script. It nonetheless reached the setup wizard. Most likely explanation: the `lethe_burner_wipe` service declares `seclabel u:r:init:s0`, which init resolves against the loaded sepolicy. Without a `lethe` domain or the `init` rules that the LETHE policy adds, init may have failed-soft on the seclabel and never started the service. With sepolicy compiled in (May-4), the seclabel resolves cleanly → service runs → wipe nukes the parent dirs. So the same script that's silent on May-2 is destructive on May-4.

The build notes' fix #5 (replace `settings put` with `rm -f settings_secure.xml`) was driven by exactly this kind of post-fs-data interaction; the parent-dir-vs-contents distinction is the same family of mistake — wholesale `rm -rf` of dirs init/installd expect to exist.

### Fix

Replace the `rm -rf /data/{app,data,user,user_de}` with a contents-only wipe that preserves the parent dirs:

```sh
for d in /data/app /data/data /data/user/0 /data/user_de/0; do
    [ -d "$d" ] && find "$d" -mindepth 1 -maxdepth 1 -exec rm -rf {} +
done
```

Applied in [scripts/runtime/lethe-burner-wipe.sh](../../scripts/runtime/lethe-burner-wipe.sh) (this commit). Also pushed to the validation device's `/system/bin/` for an immediate boot test before another full rebuild (TWRP `mount -o remount,rw /system && adb push`). The /data/data, /data/user_de/0, /data/app parent dirs were also re-created on the device with proper ownership + restorecon since the failed boot left them gone.

If this boot succeeds, the source fix needs to ride in a clean rebuild before any release artifact is uploaded — the OTA at `Lethe-1.0.0-t0lte.zip` still has the broken wipe script.
