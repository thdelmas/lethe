# LETHE v1.0.0 — Build & Validation Notes

Captured during the May 1-2, 2026 build/validation push for cm-14.1 t0lte (Galaxy Note II LTE). Pin this doc — most of these gotchas burned hours and are not obvious from the code.

## Most important operational note

**Note II first-boot dexopt takes 25–30 minutes.** During that window the device sits on the LineageOS bootlogo with no progress indicator and ADB shows nothing or `unauthorized`. This looks **identical to a bootloop**. Do not call failure before 30 minutes wall-clock from the reboot. The watchdog cap was 10–15 min on early test runs and we wrongly diagnosed a half-dozen successful boots as failures, then spent hours chasing fixes for non-bugs.

After first boot, subsequent boots are 30–60 seconds.

## Build pipeline real bugs fixed today

These are committed (see `git log` from 2026-05-01..02). Listed for future reference / new device codenames.

1. **`apply-overlays.sh` was silently shipping nothing.** `cp file system/bin/foo` in the LineageOS source tree is **not** how AOSP ships files — files have to be declared via `Android.mk` modules or `PRODUCT_COPY_FILES`. Every overlay run "succeeded" with a 254 MB OTA that contained no LETHE binaries, scripts, init files, or configs. Fixed in [73d507d](../../) (`add_to_system` helper + PRODUCT_COPY_FILES).
2. **8 LETHE prop keys exceeded Android's 31-byte `PROPERTY_KEY_MAX`** and broke `post_process_props.py`. Renamed to abbreviated forms in [84a511d](../../). Compatible with what was already on devices.
3. **`ro.build.display.id=LETHE\ 1.0.0`** — backslash-space split across echo statements, produced a 236-byte mangled value (limit: 91). Switched to `LETHE-1.0.0`.
4. **cm-14.1 still uses `vendor/cm/`**, not `vendor/lineage/`. apply-overlays.sh hardcoded the new path; props were silently skipped on legacy devices. Fixed in [ccd0b8c](../../).
5. **eSIM wipe declared on but unimplemented.** Manifest said `esim_profiles: true`, runtime overlay said `false` with explanatory comment. Aligned both to the reality (deferred — needs EuiccManager API path).
6. **No README at repo root.** Every announcement points to GitHub; visitors landed on a nameless repo.
7. **Bios/PreuJust framing in release note.** Billed as shipping modules; in v1.0 they're agent-boundary topics only.

## v1.0.x source-side issues that broke t0lte boot

Each of these caused at least one bootloop or zygote crash-loop:

1. **Hyphens in init service names.** cm-14.1's init parser rejects `lethe-burner-wipe` etc. as invalid service names. Renamed to underscores (`lethe_burner_wipe`).
2. **`capabilities` keyword in init.rc is Android 8+.** Removed from `init.lethe-tor.rc` — caps inherited via `seclabel u:r:init:s0` instead.
3. **`ro.crypto.state=encrypted` in privacy-defaults.conf crashed vold (signal 11).** On a freshly-formatted /data, vold tries to decrypt with no key and segfaults. Manifest already flagged encryption disabled per-device; the global override violated that. Removed.
4. **`ro.adb.secure=1` + `ro.debuggable=0` on a `userdebug` build crash-looped zygote.** Forcing user-build hardening on a userdebug variant put the system in a contradictory state. Build variant should drive these props, not an overlay. Deferred to v1.1.
5. **`settings put secure android_id` at post-fs-data deadlocked init.** The `settings` CLI talks to system_server's ContentProvider via binder; system_server isn't up at post-fs-data. Replaced with `rm -f /data/system/users/0/settings_secure.xml` (Android regenerates the ID on boot).
6. **Removing Trebuchet without shipping a replacement crashed system_server.** apply-overlays.sh deleted the LineageOS launcher in anticipation of Void launcher (deferred to v1.1). With no Home activity, system_server crash-loops zygote. v1.0 keeps Trebuchet.
7. **`persist.lethe.burner.enabled=true` was only in burner-mode.conf**, which ships as a static `/system/extras/lethe/` file — not loaded as build.prop. The wipe script's `getprop` returned empty and the script exited early; burner mode never ran. Added the prop to privacy-defaults.conf so the build.prop default is "true".

## cm-14.1 + Ubuntu build-environment gotchas

cm-14.1 was designed for Ubuntu 14.04/16.04 with Java 7/8 + Python 2 as system defaults. Modern Ubuntu (24.04, Java 17, Python 3) breaks immediately on:

- `jarjar.jar` builds with empty class files under Java 17 javac
- `diff_package_overlays.py` and other build scripts use Python 2-only `print` statements
- The flex prebuilt asserts in glibc locale code on newer locale archives

Mitigation: build inside `Dockerfile.cm14-build` (Ubuntu 18.04 + JDK8 + Python 2.7). Required runtime envs:

```
LC_ALL=C LANG=C            # dodges flex locale assertion
USER=root HOME=/root       # Jack-admin needs $USER set
JACK_SERVER_VM_ARGUMENTS="-Dfile.encoding=UTF-8 -XX:+TieredCompilation -Xmx4g"
                           # default Jack heap (1.5–2 GB) OOMs on framework
```

Plus the Dockerfile patches `/etc/java-8-openjdk/security/java.security` to re-enable TLS 1.0/1.1 (modern openjdk-8 disables them; Jack uses TLS 1.0 for its localhost RPC).

## chromium-webview LFS gotcha

Default `repo init -c` excludes the `pdk` group, which is where `external/chromium-webview/prebuilt/arm/` lives in cm-14.1. Without it, system_server can't load `com.android.webview` and crash-loops at runtime (build still completes — webview is a loose dependency).

Fix: `repo init -g default,pdk ... && repo sync external/chromium-webview/patches external/chromium-webview/prebuilt/arm`. Then `cd external/chromium-webview/prebuilt/arm && git lfs install --local && git lfs pull` to fetch the actual 83 MB `webview.apk` (the repo only contains LFS pointers).

## Sideload pattern

For t0lte specifically, our installed TWRP reports `ro.product.device=t03g` (3G variant) even though the running LineageOS reports `t0lte` (LTE). The OTA's updater-script asserts the t0lte family and aborts in TWRP. Workaround:

```bash
# Strip the device assertion (first line of updater-script), re-zip, re-sign:
unzip -q OTA.zip -d unsigned
sed -i '1d' unsigned/META-INF/com/google/android/updater-script
rm -f unsigned/META-INF/{MANIFEST.MF,CERT.SF,CERT.RSA,CERT.EC}
(cd unsigned && zip -qr ../unsigned.zip .)
java -Djava.library.path=<lineage>/out/host/linux-x86/lib64 \
  -jar <lineage>/out/host/linux-x86/framework/signapk.jar -w \
  <lineage>/build/target/product/security/testkey.x509.pem \
  <lineage>/build/target/product/security/testkey.pk8 \
  unsigned.zip OTA-noassert.zip
```

`signapk.jar` needs `-Djava.library.path` pointing at `libconscrypt_openjdk_jni.so` from the build's host output; without it Java throws `UnsatisfiedLinkError` on conscrypt.

## otasigcheck on freshly-formatted /data

LineageOS's `otasigcheck.sh` inside the OTA refuses to install when the new build's signing key doesn't match the device's previously-installed key. Our auto-generated test-keys differ from build to build. Workaround: TWRP **Wipe → Format Data** removes `/data/system/packages.xml`; the script's first check is `[ -f /data/system/packages.xml ]` — missing file → exit 0 → install proceeds.

## Sideload SIGBUS retries

`adb sideload` occasionally fails with `Updater process ended with signal: 7` (SIGBUS) and `Install took 2 seconds`. This appears to be flakiness in the TWRP/adbd interaction on this device. Pattern: retry the same OTA with `adb shell twrp sideload` to re-enter sideload mode, push again. Usually succeeds on the second attempt.

## Things still pending v1.0 → v1.1

These were de-scoped from v1.0 mid-flight and need to come back in v1.1:

- LETHE Agent (Rust backend + WebView wrapper system_app + Java sources) — needs proper Android.mk integration for the system_app + native binary
- Void launcher — needs APK build glue
- Dead Man's Switch (init scripts + state machine + duress PIN flow)
- IPFS OTA (binary integration + signed manifest pipeline)
- libp2p peer inference + EdgeVPN cluster (need native binaries)
- Mesh BLE preview (already labeled preview)
- Panic wipe key handler (5× power)
- ADB hardening (paired-host RSA whitelist) — needs to be driven by the build variant, not a privacy-defaults.conf override

The `apply-overlays.sh` v1.1 sections are already stubbed with `"deferred to v1.1, not packaged."` echos so re-enabling them is mechanical.

## What to NOT change going forward

- Don't switch service names back to hyphens. cm-14.1 init won't parse them.
- Don't add `capabilities` to any cm-14.1 init.rc. Use seclabel + the inherited init capabilities instead.
- Don't override `ro.crypto.state` from privacy-defaults.conf. Per-device build config drives it.
- Don't override `ro.adb.secure` / `ro.debuggable` from a global overlay. They're build-variant properties.
- Don't delete `packages/apps/Trebuchet` until Void launcher actually ships an APK in the same OTA.
- Don't run `settings put` from a `class main` / `oneshot` init service. Defer to a `late_start` service after `sys.boot_completed=1`.

## 2026-05-08 — fresh-tree rebuild + sideload-pipeline gotchas

A fresh `repo init`/sync of cm-14.1 in `~/android/lineage-v1.0.1` built a clean v1.0 OTA in 33:42 wall (warm ccache primed for ~5–10 min rebuilds going forward). Hardware-side validation didn't pan out; full diagnosis lives in [v1.0.0-flash-investigation.md](v1.0.0-flash-investigation.md). What's worth carrying forward in the build pipeline:

### Hidden hand-edits that aren't in `apply-overlays.sh`

- `build/target/product/core_minimal.mk` — `webview` removed from `PRODUCT_PACKAGES`. Workaround for the chromium-webview prebuilts being LFS-only and not auto-fetched by `repo sync`. Not in `apply-overlays.sh`. Either fold the edit in or `git lfs pull` the prebuilts on the fresh tree before `mka`. Without it the fresh tree's build dies on missing `webview`.
- `vendor/samsung/{smdk4412-common,t0lte}` — proprietary blobs, not git-tracked, not in any public repo. `extract-files.sh` requires the device booted to system to regenerate. Until then, copy `~/android/lineage/vendor/samsung/` (15 MB) into any new tree.
- `.repo/local_manifests/t0lte.xml` — copy across; default cm-14.1 manifest doesn't include the t0lte device tree.

### `Dockerfile.cm14-build` env quirks (currently passed via `docker run -e …`, not baked in)

- `LC_ALL=C LANG=C` — cm-14.1's `prebuilts/misc/linux-x86/flex/flex-2.5.39` aborts on modern glibc with `_nl_intern_locale_data` assert under `en_US.UTF-8`. The Dockerfile sets the wrong locale.
- `USER=root HOME=/root` — Jack server reads `$USER` and crashes with `unbound variable: USER` if the docker container isn't started by an interactive shell.
- `GIT_CONFIG_COUNT=1 GIT_CONFIG_KEY_0=safe.directory GIT_CONFIG_VALUE_0='*'` — silences `repo`'s "dubious ownership" warnings on `.repo/manifests` when the host UID and container UID don't match.

These should land in the Dockerfile + `build-all.sh::build_in_docker()` in v1.0.x.

### `scripts/lethe-resign.sh` gotchas

- **`-Djava.library.path=$LINEAGE_TREE/prebuilts/sdk/tools/linux/lib64` is required** on the `signapk` invocation. Outside the build, signapk dies on `UnsatisfiedLinkError: no conscrypt_openjdk_jni`. Inside `mka bacon`, the build's own signapk wrapper sets the path. Fixed in `09d90c6`.
- **`/usr/bin/zip` (Info-ZIP) silently fails when invoked from a Make recipe** under `.ONESHELL + SHELLFLAGS=-euo pipefail -c`. Emits `zip warning: name not matched: <archive>` followed by `updating:` lines for every input file, exits rc=0, leaves no archive on disk. Reproduces with positional args, `zip -@` stdin, absolute and relative archive paths. Same script via `bash -euo pipefail -c` works fine. Root cause unidentified; fixed in `adced25` by replacing the repack step with Python's `zipfile` module. **Don't put `/usr/bin/zip` back in this script.**

### New Makefile targets for hypothesis testing without rebuilding

- `make persist-set KEY=… VAL=…` / `persist-clear` / `persist-list` — write `/data/property/persist.<key>` overrides from TWRP; init picks them up at post-fs and they override `build.prop` defaults. Lets you flip a single property between flashes without touching the build.
- `make wipe-system` / `make wipe-cache` — the standalone partition wipes that `wipe-data` already had. Use `wipe-system` to force a non-patching reflash when block-OTA install lands on top of corrupt /system blocks.
- `make fresh-boot-test ZIP=… [WIPE_SYSTEM=1] [WIPE_CACHE=1] [PERSIST_KEY=… PERSIST_VAL=…]` — full clean-room cycle: wipe → resign+sideload → optional persist seed → reboot → wait-boot. Each phase prints UTC `HH:MMZ` and a worst-case ETA. Worst-case ~50 min for an OTA-sized zip on Note II.
